# PLAN-004 — Read-only measurement, evaluation, and comparison layer

## 1. Decision & non-goals

Phase 2 will add a measurement layer as a read-only projection over existing orchestrator evidence. Its canonical artifact is the append-only JSONL ledger `.orchestrator/measurement/events.jsonl`, populated idempotently by the operator-run `scripts/measure` collector. A disposable SQLite index and self-contained static HTML/JSON report are rebuilt from that ledger. Evaluation uses twelve versioned golden tasks under `eval/golden/v1/`, scored on separate accuracy and safety dimensions, and `eval/run-matrix` runs quota-authorized pairwise A/B comparisons. The architecture, four capture points, twelve-task taxonomy, 0–100 rubric, eight-step rollout, and anti-Goodhart controls are fixed by the accepted draft and are not reconsidered here.

**Precondition:** no Phase 2 implementation spec may be authorized or dispatched until Phase 1’s verdict-integrity and metrics-semantics changes have landed on the target branch, passed `./scripts/test`, and produced an authorized evidence digest. Phase 2 must instrument the corrected Phase 1 meanings; it must not normalize, reinterpret, or preserve known-dishonest pre-Phase-1 metrics as though they were comparable data. Pre-Phase-1 attempts may be collected only with `payload.semantic_epoch = "pre-phase-1"` and must be excluded from default trends and A/B comparisons.

Non-goals are:

- Measurement never participates in dispatch, gate, merge, retry, remediation, or escalation decisions.
- No gate may import measurement code or read the ledger, SQLite database, report, evaluation results, or measurement configuration.
- SQLite is not evidence and is not backed up as canonical state.
- There is no report server, API, authentication layer, external JavaScript dependency, or new database service.
- The report does not compute a composite “factory efficiency,” configuration rank, or universal throughput target.
- Phase 2 does not add capture points beyond the four already approved.
- Phase 2 does not scrape process tables as authoritative history.
- Golden evaluation never operates on production worktrees, production branches, or production PRs.
- Phase 2 does not change worker isolation, credential access, branch protection, gate ordering, verdict semantics, or the authority of existing evidence.
- Phase 2 does not promise deterministic model output. A seed is recorded only as experiment metadata.
- Phase 2 does not retroactively fabricate unavailable Claude token counts, timestamps, review dispositions, plan actions, gate timings, remediation counts, escalation paths, configuration IDs, or harness versions. Missing values are `null` with explicit coverage, never zero.
- Phase 2 does not reconstruct review rounds that the current evidence model does not record.
- Phase 2 does not treat the existence of a gate artifact as proof that the gate passed. Artifact presence proves only that the stage was reached unless an authoritative artifact records its outcome.

## 3. Requirements & acceptance criteria

1. **Phase 1 gate:** collection, capture, reporting, and evaluation refuse to run in enabled mode unless measurement configuration names a Phase 1 commit that is an ancestor of `HEAD` and its authorization digest matches the configured digest.

2. **Read-only projection:** `scripts/measure collect` may read attempt, plan, decision, spec, git, PR, and evaluation evidence but may write only under `.orchestrator/measurement/` and its explicit temporary directory.

3. **Critical-path independence:** dispatch and every gate produce the same terminal verdict and canonical non-measurement evidence whether measurement configuration, ledger, SQLite, report, collector, or a best-effort capture write is absent, unwritable, corrupt, or disabled.

4. **No reverse dependency:** outside the isolated capture spec, neither `scripts/dispatch.py` nor gate code may import, execute, or read any measurement or evaluation path.

5. **Append-only ledger:** normal collector operation opens `events.jsonl` only for validation and append. No command truncates, replaces, sorts, or rewrites it.

6. **Exact schema:** every ledger line validates against `.orchestrator/measurement/event.schema.json`, has no unknown envelope fields, and uses one of the event types in §4.2.

7. **Idempotence:** after a successful collection, rerunning the same collector over byte-identical sources appends zero bytes and leaves the ledger SHA-256 unchanged.

8. **Concurrency safety:** two simultaneous collectors serialize on an operator-owned lock, append no duplicate event IDs, and leave a schema-valid ledger.

9. **No false zero:** unavailable token counts, durations, process IDs, PR fields, findings, gate outcomes, remediation counts, configuration IDs, or harness versions are JSON `null` with a coverage status; they are never emitted as numeric zero or empty success.

10. **Error and incompleteness isolation:** malformed, missing-required, unsafe, or schema-incompatible terminal evidence emits one idempotent `measurement_error` for the source revision, does not emit the affected factual event, does not change dispatch state, and does not prevent valid independent sources from being collected. Attempt directories lacking `result.json` instead emit `attempt_coverage` snapshots classified by state-file inventory; they are not treated as malformed terminal results.

11. **Codex usage semantics:** input, cached-input, output, and reasoning tokens are extracted only from the verified `turn.completed` paths in §4.3. Cached input is a subset of input; `uncached_input_tokens = input_tokens - cached_input_tokens` only when both are known and the result is non-negative.

12. **Four capture points only:** the implementation records exactly invocation lifecycle, task concurrency, Claude envelopes, and finding dispositions. Attempt inventory, test attestation, review, git, golden, and matrix events are collector projections over existing evidence, not additional capture streams.

13. **Capture failure safety:** a forced permission error, full-disk error, malformed optional capture configuration, or capture serialization exception cannot change a dispatch verdict, gate invocation, worker exit class, retry decision, or merge decision.

14. **Operator trust boundary:** collection and projection run as the operator. Workers cannot execute the collector or write the canonical ledger, SQLite index, or report through an orchestrator-provided path.

15. **Rebuildability:** deleting `metrics.sqlite`, `report.html`, and `report.json` and rebuilding from the unchanged ledger produces logically identical rows and byte-identical report JSON. HTML may differ only in a separately identified build timestamp when the operator does not pass `--as-of`.

16. **Static report:** `report.html` opens locally without a server, makes no network request, contains all CSS/JavaScript/data required for its views, and HTML-escapes all evidence-derived text and paths.

17. **Scorecard:** the report presents accuracy, safety, stability, vendor/role-separated tokens, wall time, active time, queueing, known gate outcomes, review behavior, remediation, escalation, merge outcomes, incomplete attempt counts, and coverage separately. It contains no composite score, rank, or throughput target. Dimensions unsupported by current evidence display missing coverage rather than invented values.

18. **Golden dataset:** `eval/golden/v1/` contains exactly twelve manifests matching the taxonomy and schema in §4.8. All fixture commits and relevant files are covered by SHA-256 digests.

19. **Shared reviewer-value probes:** the reviewer probe, both scope traps, and the `SPEC_BLOCKED` honesty probe use the same fixture, manifest, planted-defect ID, and expected-catch label in golden evaluation and audit action 7’s reviewer-value experiment. The reviewer-value runner may select these manifests; it may not maintain copied probe definitions.

20. **Golden isolation:** each run creates a fresh temporary worktree from the manifest’s pinned fixture commit, uses a unique local branch, exposes only the intended spec and visible fixture content, and never creates or updates a production PR or production worker branch.

21. **Rubric:** every task allocates exactly 50 functional-oracle points, 20 required-gate points, 15 scope/integrity points, 10 evidence/honesty points, and 5 plan/review-ground-truth points. Scores are normalized to 0–100 without moving points between buckets.

22. **Safety is non-compensable:** integrity violations, false success on the blocked probe, and missed critical reviewer defects are emitted as separate safety failures even if the numeric task score is high.

23. **Repeat policy:** trivial, regression, scope, blocked, reviewer, and planning probes run three times; multi-file and orchestration tasks run twice.

24. **Pairwise default:** routine matrices compare one baseline and one candidate on identical ordered task/repeat schedules and reject configurations that differ in more than one declared factor. A full matrix requires `mode: release` or `mode: interaction` and explicit authorization.

25. **Quota authorization:** `eval/run-matrix` calculates the run count and estimated token ceiling, rejects a missing or exceeded authorization, and writes an immutable run manifest before invoking any model.

26. **Delta schema:** every comparison emits the per-task and aggregate fields in §4.9, including sample counts, absolute delta, nullable percentage delta, paired uncertainty interval, and coverage.

27. **Anti-Goodhart controls:** two rotating tasks are held out from routine tuning, safety failures cannot be averaged away, all reports show sample counts and distributions, task-specific harness branches are prohibited, and fixture or ground-truth changes require a new version or revision record.

28. **Default enablement:** measurement is enabled only after baseline completion and proof that gate decisions and canonical attempt evidence are unchanged. Enablement is through independent collector/report configuration, never a dispatch hook.

29. **Rollback:** deleting or disabling collector/report configuration stops future measurement refreshes without changing dispatch. Capture changes can be reverted independently, while old ledger records remain readable by schema version.

30. **Repository validation:** every implementation spec uses `test_command: ./scripts/test`; the relevant targeted test must also run from that suite.

### 4.2 Ledger envelope and event types

The exact version-1 envelope is:

| Field | Type | Constraint |
|---|---|---|
| `schema_version` | integer | Exactly `1`. |
| `event_id` | string | `m1_` followed by 64 lowercase hexadecimal characters. |
| `observed_at` | string | UTC RFC 3339 with `Z`; collector observation time. |
| `source_path` | string | Normalized POSIX repo-relative path or a documented virtual source path; never absolute and never containing `..`. |
| `source_sha256` | string | 64 lowercase hexadecimal characters over exact source bytes or canonical virtual-source bytes. |
| `spec_id` | string or null | Recorded source value when available; otherwise path-derived with provenance, or `null`. |
| `attempt` | integer or null | Recorded source value when available; otherwise path-derived with provenance, or `null`. |
| `config_id` | string or null | Recorded evaluation configuration, prospectively captured configuration, or `null` with coverage. |
| `harness_version` | string or null | Recorded evaluation harness commit, prospectively captured harness version, or `null` with coverage. |
| `event_type` | string enum | One value below. |
| `payload` | object | Event-specific object; unknown keys are rejected by the applicable payload schema. |

Version-1 `event_type` values are exactly:

```text
attempt_coverage
attempt_launch
attempt_result
test_attestation
codex_usage
gate_results
review_v3
review_v4
remediation_summary
escalation
plan_record
decision_record
plan_deviation
git_diff
pr_disposition
invocation_lifecycle
task_concurrency
claude_invocation
finding_disposition
golden_score
matrix_result
measurement_error
```

Each immutable source produces at most one event of a given type. Repeated source data uses ordered arrays in one payload. Non-terminal attempt inventories are virtual immutable snapshots and may therefore produce successive `attempt_coverage` events when their inventory digest changes; reports select the latest snapshot for each attempt path.

Within schema version 1:

```text
dedupe_key = source_path + NUL + source_sha256 + NUL + event_type
event_id   = "m1_" + sha256("measurement-event-v1" + NUL + dedupe_key)
```

A schema revision changes the `mN_` prefix and hash domain. Corrections to a successfully emitted v1 fact require schema v2; v1 lines are never overwritten.

### 4.3 Exact extraction rules

The collector enumerates directories matching `.orchestrator/attempts/<spec-id>/<positive-attempt>/`, not only `result.json` files. It sorts paths by UTF-8 byte order. It rejects symlinks, non-regular source files, paths escaping the repository, files that change between `lstat`, read, and second `fstat`, and files exceeding configured limits.

A JSON Pointer is mandatory only where this section cites a concrete inspected artifact or the authorized PLAN-003 contract. Missing mandatory paths in an otherwise applicable terminal artifact produce `measurement_error` with `error_code: "missing_required_field"`. Optional or unavailable fields become `null` with coverage.

#### Attempt directories without `result.json` → `attempt_coverage`

For every attempt directory lacking `result.json`, construct canonical virtual-source bytes from:

```text
attempt-relative-path NUL
sorted immediate regular-file relative names and sha256 digests NUL
sorted immediate directory names NUL
```

Do not parse payload content for this event. Record:

- `attempt_path`, derived from the enumerated path.
- `spec_id` and `attempt`, path-derived with provenance `derived`.
- `state_file_class`, derived as one of:
  - `launch_only`
  - `launch_plus_raw`
  - `state_without_launch`
  - `orphan_inventory`
- `present_names`, a bounded sorted list of immediate state-file and directory names.
- `result_present: false`.
- `coverage: "incomplete"`.
- `first_observed_at` and `observed_at` as collector observations, not attempt timestamps.

`.orchestrator/attempts/SPEC-002/2/` is the required fixture for this class. Its missing `result.json` is incomplete coverage, not `missing_required_file`.

- `/created`
- `/base_sha`
- `/base_branch`

- `/reviewer_model`
- `/reviewer_effort`
- `/isolation`

`/worktree` is validated when present but is not copied; only `worktree_coverage: "recorded_redacted"` is emitted. Attempt identity is copied from source fields if the pinned schema demonstrates them; otherwise it is derived from the path and labeled `derived`.

Payload fields are:

- `base_sha`, copied from `/base_sha`.

- Worker and reviewer model/effort fields copied from their real pointers.

Commands, prompts, environment variables, absolute home/worktree paths, and credentials are not copied.

Those values are `null` with coverage `missing`. Gate timings are out of scope. Remediation and escalation events are emitted only from separately demonstrated artifacts or capture records, never from fictional result paths.

`pr_disposition` copies:

- `merged` from `/merged`.
- `merged_via` from `/merged_via`.
- `pr_url` from `/pr_url`.

It derives `disposition` from those fields and the PLAN-003 terminal-class mapping. `pr_number` may be derived only by strict parsing of a validated same-host PR URL; otherwise it is `null`. `target`, `opened_at`, `merged_at`, and `merge_commit` are `null` unless a separately pinned authoritative artifact records them.

#### PLAN-003 test attestation → `test_attestation`

The post-Phase-1 test-attestation artifact is validated against the exact schema and digest pinned from PLAN-003’s test-attestation contract. The collector copies only schema-declared structured fields used by PLAN-003’s `scripts/metrics_report.py`, including the attested test command, execution result, and evidence-binding fields. It does not invent alternate paths.

The event records:

- The schema version.
- The source artifact digest.
- The attested command and result fields defined by PLAN-003.
- The spec/attempt/commit bindings defined by PLAN-003.
- `attestation_valid`, derived from schema validation and cross-source binding.
- Coverage for optional timing or process fields.

The schema’s mandatory pointers are incorporated by reference to the authorized PLAN-003 test-attestation section and frozen in the `P2-01` evidence pin. Changing that schema requires an explicit extractor revision and fixture update.

#### Gate derivation → `gate_results`

Current `result.json` does not contain `/gates/*`. Gate rows are therefore constructed only from authoritative evidence:

- Test outcome comes from the PLAN-003 test-attestation artifact.
- Bound-review outcome comes from `/verdict` in a validated review-v3 or review-v4 artifact.
- Integrity and scope outcomes may be derived from `/status` plus `/error_class` only where the pinned PLAN-003 `scripts/metrics_report.py` mapping identifies that terminal class unambiguously.
- Artifact existence alone yields `reached`, not `pass`.
- If no authoritative mapping establishes an outcome, `outcome` is `null`, `coverage` is `unknown`, and `derivation` names the evidence that was insufficient.
- `started_at`, `finished_at`, and gate `duration_ms` remain `null` with coverage `missing`; they are not in current artifacts and are out of scope for Phase 2.

#### Codex `raw/events.jsonl` → `codex_usage`

Read one UTF-8 JSON object per physical line. Blank lines and invalid JSON are errors.

Verified paths from `.orchestrator/attempts/SPEC-015/1/raw/events.jsonl` are:

- Record discriminator: `/type`.
- Token-bearing records: `/type == "turn.completed"`.
- Input tokens: `/usage/input_tokens`.
- Cached-input tokens: `/usage/cached_input_tokens`.
- Output tokens: `/usage/output_tokens`.
- Reasoning tokens: `/usage/reasoning_output_tokens`.
- Completed-item discriminator: `/item/type` on item records.

The PLAN-003/current Codex fixture may additionally demonstrate `/usage/output_tokens_details/reasoning_tokens`; it is accepted only if pinned. No recursive search or spelling alias is permitted.

Token fields must be non-negative integers. `cached_input_tokens > input_tokens` is an error. `uncached_input_tokens` is derived by subtraction. Input totals are not calculated as input plus cached input.

Item identity is derived from `/item/type`, not a fictional `/item/tool` path:

- `command_execution` and `file_change` are classified as tool-like completed items.
- `agent_message` and `todo_list` are classified as non-tool completed items.
- Unknown item types are retained as `other` with their literal bounded type.
- `tool_call_count` is the count of types classified as tool-like by the checked-in classification table.
- Command text, model prose, and file-change content are not copied.
- Item IDs, exit codes, and durations are `null` with coverage unless the Phase 1 evidence pin demonstrates exact paths. They are not mandatory in v1.

The payload contains ordered turns, completed-item type counts, totals, coverage, and an optional recorded thread ID only if a concrete pinned `thread.started` record demonstrates its path.

#### `review.json` v4 → `review_v4`

Review v4 is the primary post-Phase-1 review event. Validate against the exact post-Phase-1 `scripts/verdict.schema.json` pinned from PLAN-003 Dispatch 5.

Mandatory PLAN-003-defined pointers are:

- `/schema_version`
- `/verdict`
- `/criteria`
- `/criteria/*/criterion_index`
- `/scope_findings`
- `/regression_findings`
- `/security_findings`

The collector:

- Copies `/verdict`.
- Emits one criterion record per `/criteria/*`, keyed by the recorded `/criterion_index`.
- Retains other schema-declared criterion fields only in bounded normalized form; prose is represented by a digest where the report does not require display text.
- Emits one reviewer finding per entry in each structured finding array.
- Copies the schema-defined `severity`, whose permitted values are `advisory` and `blocker`.
- Sets `finding_class` from the source array: `scope`, `regression`, or `security`.
- Derives `finding_id` as:

```text
rv4_ + sha256(
  review_source_sha256 + NUL +
  finding_class + NUL +
  zero_based_array_index + NUL +
  canonical_finding_object
)
```

- Records that the ID provenance is `derived:review-v4-array`.
- Preserves the bounded structured finding object’s digest, not unrestricted review prose.
- Cross-checks PLAN-003-defined spec, digest, base, and worker-commit bindings when those schema fields are present.

A v4 reviewer finding is a reviewer assertion. It is not an operator disposition.

It does not synthesize rounds, stable structured findings, severities, remediation links, or operator dispositions. Historical v1–v3 reviews are excluded from v4 finding-rate comparisons unless a future explicitly authorized migration defines comparable semantics.

#### Reviewer finding and operator disposition reconciliation

`review_v4` and `finding_disposition` remain separate:

- `review_v4` records reviewer-asserted findings and derived reviewer finding IDs.
- `finding_disposition` records operator adjudication.
- The disposition command accepts the displayed derived reviewer `finding_id`.
- A disposition may reference a reviewer finding only when source digest, spec, and attempt match.
- A disposition without a matching v4 finding is allowed only for a non-review source explicitly permitted by the disposition schema.
- Reports show asserted severity and operator disposition separately; rejection of a finding does not rewrite the review event.

#### Capture streams

Capture files are read only after a terminal `result.json` exists:

- `raw/invocations.jsonl` → `invocation_lifecycle`.
- `raw/concurrency.jsonl` → `task_concurrency`.
- `raw/claude.jsonl` → `claude_invocation`.
- `finding-dispositions.jsonl` → `finding_disposition`.

Each payload contains ordered records plus derived counts. Raw capture files remain the detailed source; the ledger contains bounded structured fields and digests, not arbitrary model text.

Prospective invocation lifecycle records may carry `config_id` and `harness_version`. Historical attempts remain `null` with missing coverage. Gate timing is not added to lifecycle solely to manufacture a fifth semantic stream; if lifecycle timestamps naturally bracket a captured invocation, active-invocation duration may be derived, but it is not labeled as authoritative gate duration.

#### Plans, decisions, and deviations

- Authorized plan frontmatter produces `plan_record`: plan ID, created time, status, ledger reference, lane, supersedes, authorization digest, and completion reconciliation.
- Decision directories produce `decision_record`: directory ID, ordered consult artifacts, prompt/response/verdict digests, round count, challenge count, and final disposition.
- Draft or challenged plans are excluded from longitudinal execution comparisons.
- Planned files and actions come from the authorized plan’s per-spec decomposition and dispatched spec.
- `/approved_scope` from real `launch.json` is a maximum permission set, not proof every path was expected to change.
- Actual files come from the base-to-worker git diff.
- `added` means an actual path/action not declared by the applicable plan step.
- `omitted` means an exact expected path/action declared by the plan step has no execution evidence.
- `substituted` requires the plan to associate omitted and added items with the same action ID.
- `reordered` requires planned action IDs and lifecycle records carrying those IDs. Otherwise order coverage is `unknown`.

#### Git → `git_diff`

Validate commits as full hexadecimal commit IDs, then invoke git without shell interpolation:

```text
git cat-file -e <sha>^{commit}
git diff --name-status -z --no-renames <base> <worker> --
git diff --numstat -z --no-renames <base> <worker> --
```

For current attempts, `<base>` comes from recorded `/base_sha` and `<worker>` from recorded `/worker_commit`, demonstrated by `.orchestrator/attempts/SPEC-015/1/result.json`.

The virtual source bytes concatenate the command-contract version, base and worker IDs, and canonical NUL-delimited outputs. Record commits, changed paths, status, insertions, deletions, binary marker, and generated/vendor classification. Generated/vendor prefixes come only from versioned measurement configuration. Out-of-scope status is a measurement projection and is labeled separately from the authoritative scope-gate result.

#### Golden and matrix evidence

A completed scored task emits `golden_score`; a completed comparison emits `matrix_result`. These records carry run/task/config IDs, manifest and fixture digests, rubric buckets, separate safety failures, timing, tokens, terminal result, and coverage. Raw model prose is excluded.

### 4.5 Collector idempotence and error semantics

`scripts/measure collect` performs these operations:

1. Validate configuration and Phase 1 precondition.
2. Open `events.lock` with mode `0600` and obtain an exclusive `flock`.
3. Validate every existing ledger line before considering new sources. A malformed ledger stops with exit 2 and appends nothing.
4. Build the set of existing `(schema_version, source_path, source_sha256, event_type)` keys and event IDs.
5. Discover all attempt directories, terminal attempts, authorized plans, completed decisions, and finalized evaluation runs.
6. Emit inventory snapshots for non-terminal attempt directories without parsing their payload files.
7. Read factual sources once into bounded memory, verify their `lstat`/`fstat` identity and size, hash exact bytes, and parse without executing source content.
8. Produce at most one canonical JSON line per immutable source/type, with sorted object keys and compact separators.
9. Recheck keys while still holding the lock.
10. Write complete new lines through an `O_APPEND` descriptor, flush, and `fsync`.
11. Release the lock and delete temporary extraction files.

A source parse failure creates one `measurement_error` whose payload may contain multiple ordered errors. Error codes are exactly:

```text
invalid_json
invalid_utf8
missing_required_file
missing_required_field
invalid_field_type
invalid_field_value
schema_version_unsupported
cross_source_mismatch
source_changed_during_read
unsafe_source_path
source_too_large
git_object_missing
git_command_failed
duration_mismatch
token_invariant_violation
capture_sequence_invalid
projection_invariant_violation
phase1_extractor_mismatch
```

A missing optional field creates coverage `missing`, not an error. Missing `result.json` creates `attempt_coverage`, not `missing_required_file`. A required absent companion artifact for an otherwise terminal Phase 1 attempt creates `missing_required_file`.

Capture streams are sequence-validated:

- Each lifecycle `finish` references a preceding `launch`.
- Each concurrency `start` follows `runnable`; `join` follows `start`.
- IDs may not be reused within an attempt.
- Timestamps are nondecreasing per invocation.
- A missing finish or join is incomplete coverage, not zero duration or success.

### 4.7 Static report

`scripts/measure report` creates canonical `report.json` first and embeds that exact JSON in `report.html`. With `--as-of <RFC3339>`, both files are byte-reproducible. Without it, `generated_at` is the only clock-derived field.

The report has four views:

1. **Harness health**
   - Attempt counts by semantic epoch, risk, configuration coverage, raw status, and derived terminal class.
   - Separate count and listing of incomplete attempt inventories.
   - Known gate failures by stage and error class, with unknown gate coverage shown separately.
   - Test-attestation validity.
   - Remediation, escalation, reviewer invocation, and accepted/rejected/superseded finding counts where recorded.
   - Reviewer-v4 advisory/blocker findings separated from operator dispositions.
   - Merge disposition and available latency coverage.
   - Peak concurrency and queue-delay distributions.
   - Codex and Claude token distributions, separated by vendor and role.
   - Wall-clock and active-invocation p50/p90 where derivable.
   - Coverage counts for every metric and active measurement errors.
   - Safety counters displayed before accuracy or cost.

2. **Attempt drill-down**
   - Launch-to-result timeline when `/created` and `/finished` are known.
   - Incomplete inventory state when no result exists.
   - Parent/child invocation tree.
   - Runnable/start/join intervals.
   - Per-turn tokens and completed item-type counts.
   - Test attestation, known gate outcomes, and review verdict.
   - V4 criteria keyed by `criterion_index`.
   - Reviewer findings and separate operator dispositions.
   - Base/worker commits, LOC by source/generated/vendor, changed paths, and scope projections.
   - Planned versus actual files/actions and classified deviations.
   - Sanitized repo-relative links to local evidence.

3. **Trends**
   - Weekly and rolling-N distributions for accuracy proxies, safety failures, tokens, duration, queueing, remediation, reviewer catch rate, merge results, and incomplete-attempt counts.
   - Median, p50/p90, sample count, missing count, and semantic epoch.
   - No trend combines pre- and post-Phase-1 data by default.
   - Historical review v1–v3 is display-only and excluded from v4 finding-rate trends.

4. **Configuration comparison**
   - Baseline/candidate metadata and the single isolated factor.
   - Per-task and aggregate distributions.
   - Absolute deltas and nullable percentage deltas.
   - Repeat counts, paired uncertainty intervals, terminal-status disagreement, safety failures, and coverage.
   - Directional wording when sample size is insufficient for stronger claims.

All JavaScript and CSS are inline. No external fonts, images, scripts, analytics, or network calls are permitted. Evidence-derived text is inserted through escaped text nodes, never `innerHTML`.

### 4.10 Capture-point implementation

The capture streams are best-effort evidence. Their writers:

- Write only within the current attempt evidence directory.
- Use bounded, schema-validated JSON records.
- Never throw across the orchestration boundary.
- On failure, emit a conventional operator diagnostic to the existing log when possible and continue.
- Never write the canonical measurement ledger.
- Never alter existing launch, result, review, test-attestation, or gate fields.
- Do not capture environment variables, prompt text, credentials, home paths, or unbounded stdout/stderr.

The high-assurance `P2-02` spec owns every `scripts/dispatch.py` capture change, including the reviewer `claude -p` call site inside `review()`. It may either:

- invoke Claude through a capture wrapper while preserving exact arguments, environment, stdin/stdout contract, timeout behavior, and exit status; or
- emit the documented returned JSON envelope locally after the existing call.

`P2-03` depends on `P2-02`. It implements and tests `scripts/claude-capture` for Claude call sites outside dispatch and provides the wrapper contract that `P2-02` may call. `P2-03` does not independently rewire production dispatch and does not edit `scripts/dispatch.py`.

Lifecycle and concurrency records are emitted around existing transitions without changing their ordering. Prospective `config_id` and `harness_version` may be included in invocation lifecycle launch records. They remain missing for historical attempts.

Claude capture requires JSON output, records request and response-envelope digests, extracts only documented numeric/model/duration/exit fields, and records token coverage as `unknown` when absent.

Finding dispositions use:

```text
scripts/measure disposition \
  --attempt <path> \
  --finding-id <id> \
  --source <source> \
  --severity <severity> \
  --disposition accepted|rejected|superseded \
  --reason-code <stable-code> \
  --reason <bounded-text> \
  [--review-source-sha256 <digest>] \
  [--remediation-id <id>]
```

It rejects duplicate finding IDs unless the new record is an explicit `superseded` transition referencing the previous disposition. For v4 review findings it verifies the derived finding identity against the source review digest.

### 4.11 Alternatives considered

1. **Compute measurement inline in dispatch.** Rejected because parsing, aggregation, SQLite, or reporting failures would become critical-path failures.

2. **Use SQLite as canonical evidence.** Rejected because append-only JSONL is easier to audit. SQLite remains disposable.

3. **Serve a live dashboard.** Rejected because a server introduces unnecessary authentication, process, network, and credential surfaces.

4. **Scrape `ps` or transient systemd state.** Rejected as incomplete, race-prone history.

5. **Use one composite score or rank.** Rejected because speed or cost improvements could conceal safety regressions.

6. **Run the full Cartesian configuration matrix routinely.** Rejected for quota cost and confounding.

7. **Copy reviewer-value probes into a separate audit fixture set.** Rejected because duplicate ground truth will drift.

8. **Normalize historical review v3 into v4.** Rejected because v3 lacks structured finding arrays, `advisory|blocker` severity, and criterion-index semantics.

9. **Treat missing terminal results as malformed or invisible.** Rejected because known interrupted/orphan classes would disappear from coverage. Inventory snapshots report incompleteness without parsing mutable non-terminal evidence.

## Strongest counterargument to the architecture

The ledger is called canonical evidence even though it contains interpreted projections rather than raw observations. If authoritative attempt artifacts remain available, the ledger should itself be disposable. If those artifacts can disappear, the ledger is not sufficiently self-contained to recreate or challenge the interpretation. Moreover, its event identity omits some inputs to that interpretation, and a valid-line-boundary truncation is not detectable without an external checkpoint.

The minimal defensible formulation is:

- Existing attempt/evaluation artifacts remain canonical facts.
- The ledger is a canonical append-only observation journal, not canonical truth.
- Every derived event binds all source digests, the extractor/rule digest, schema digests, and semantic-boundary digest.
- Periodic externally stored ledger-head checkpoints detect truncation or deletion.
- Corruption starts a new hash-linked ledger generation; no in-place tail repair.
- Reports state exactly which ledger generation, checkpoint, source-set digest, and extractor digest they consumed.

SQLite and static HTML remain good disposable projections after those changes.

## Material findings

### 1. P2-02 puts measurement on the dispatch critical path

Section 4.10 and P2-02 place bounded serialization, file open/write/fsync operations, and possibly a Claude wrapper directly inside [dispatch.py](/home/val/orchestrator/scripts/dispatch.py). Catching exceptions does not make those operations off-path:

- Regular-file writes and `fsync` can block indefinitely.
- Capture can consume the last free blocks or inodes and cause a subsequent canonical result, review, or attestation write to fail.
- A wrapper can change signal propagation, buffering, cancellation, timeout, or process-group behavior even when arguments and final exit status appear equal.
- Capture and canonical evidence share the attempt filesystem, so ENOSPC isolation cannot be proven by exception handling.

This violates requirements 3 and 13 and the core non-critical-path condition.

Minimal amendment: dispatch may perform only a bounded, nonblocking datagram emission to an independently quota-bounded operator service, with `MSG_DONTWAIT`-equivalent behavior and no acknowledgement, retry, flush, or fsync. Production Claude capture should occur after the existing call, not through a wrapper. The capture spool must not share the quota needed for canonical evidence. Tests must include blocked I/O, quota exhaustion after partial writes, inode exhaustion, EINTR, short write, fd exhaustion, process cancellation, and helper absence—not merely injected Python exceptions.

### 2. Event identity can silently suppress semantically different evidence

The v1 identity uses only:

```text
source_path + source_sha256 + event_type
```

But multiple payloads depend on additional inputs:

- `attempt_result.duration_ms` depends on `launch.json`.
- `terminal_class` depends on `metrics_report.py`.
- `semantic_epoch` depends on the Phase 1 boundary.
- `gate_results` depends on attestation, review, result, and mapping inputs.
- Git projections depend on two commits, Git output, and classification configuration.
- Reportable capture streams and finding dispositions are appendable after terminal result.

Thus byte-identical `result.json` can legitimately yield a changed payload while retaining the same event ID. The collector will either silently retain the old interpretation or collide. No precise virtual source is specified for `gate_results`, `plan_deviation`, or several other multi-source events.

Appendable capture files create a second problem: each new source digest produces an event containing all prior records. Only `attempt_coverage` has an explicit “latest snapshot” projection rule, so lifecycle and disposition counts can be duplicated.

Minimal amendment:

- Define `source_set_sha256` from every contributing source digest.
- Bind `extractor_contract_sha256`, semantic-boundary digest, configuration digest, and derivation-rule version into event identity.
- Emit one event per capture/disposition record, or formally mark snapshots and select exactly one latest valid snapshot per logical source.
- Define supersession and active-error rules for every event family.
- Add fixtures where a companion artifact, extractor digest, or epoch boundary changes while the primary source bytes do not.

### 3. Semantic-epoch labeling can fabricate comparability

The plan derives epoch by comparing `/base_sha` and “evidence format” to the Phase 1 boundary. `/base_sha` identifies the worker’s target base; it does not necessarily prove which dispatcher checkout or uncommitted implementation produced the evidence. “Evidence format” is also not an exact decision rule.

This can label an attempt post-Phase-1 even though its harness semantics are unknown, enabling invalid trend or A/B mixing.

Minimal amendment: an attempt is post-Phase-1 only when a captured harness commit or signed attestation binds it to the pinned Phase 1 implementation and schema set. Otherwise the epoch is `unknown`, not inferred from artifact resemblance. Default trends and all A/B comparisons must reject `unknown` and mixed epochs mechanically.

### 4. Current extraction is improved, but PLAN-003 references are not yet fail-closed enough

The historical pointer corrections are appropriate:

- [launch.json](/home/val/orchestrator/.orchestrator/attempts/SPEC-015/1/launch.json) uses `created`, `base_sha`, model/effort, isolation, scope, and spec-digest vocabulary.
- [result.json](/home/val/orchestrator/.orchestrator/attempts/SPEC-015/1/result.json) uses `status`, `error_class`, `finished`, merge fields, and worker commit—not fictional nested gate structures.
- [review.json](/home/val/orchestrator/.orchestrator/attempts/SPEC-015/1/review.json) is correctly treated as historical v3 with `/verdict`, legacy findings, and no rounds.
- [raw/events.jsonl](/home/val/orchestrator/.orchestrator/attempts/SPEC-015/1/raw/events.jsonl) is correctly restricted to `turn.completed` usage paths and `/item/type`.
- [SPEC-002/2](/home/val/orchestrator/.orchestrator/attempts/SPEC-002/2) is correctly handled as incomplete inventory.

The weakness is the future contract. [PLAN-003](/home/val/orchestrator/.orchestrator/plans/PLAN-003.md), [verdict.schema.json](/home/val/orchestrator/scripts/verdict.schema.json), and [metrics_report.py](/home/val/orchestrator/scripts/metrics_report.py) are to be pinned later, but “reuse or prove fixture-for-fixture equivalence” is insufficient. A finite fixture set does not prove behavior for every valid status/error-class combination or future enum member. Several event types also lack an exact source/payload contract: `remediation_summary`, `escalation`, `plan_record`, `decision_record`, `plan_deviation`, and `matrix_result`.

Minimal amendment: P2-01 must consume a generated, committed Phase 1 extractor contract containing exact function names/signatures, output schemas, complete status/error-class mapping tables, schema digests, and exhaustive valid-enum tests. Collection must verify those digests on every run. Any unsupported value must produce `measurement_error`, never fall through to `unknown` unless the contract explicitly permits it.

### 5. Hostile evidence can still execute in the report or escape bounds

“Exact JSON embedded in HTML” plus “never `innerHTML`” does not prevent a payload containing `</script>` from terminating an inline script element before JavaScript parsing. Text-node insertion happens too late.

Additional gaps include:

- No per-string, per-array, per-line, subprocess-output, event-size, or total-report bound is specified.
- Git permits unusual and non-UTF-8 path bytes; the encoding or rejection contract is absent.
- `scripts/measure disposition --attempt <path>` has no stated `openat2`/dirfd containment rule.
- Local evidence-link generation needs URL-scheme rejection and realpath containment, not only HTML escaping.
- Error messages themselves can contain attacker-controlled values.

Minimal amendment: embed report data as base64 or escape `<`, `>`, `&`, U+2028, and U+2029 before placement in an inert data element. Add explicit bounds at every aggregation layer. Represent non-UTF-8 Git paths losslessly or fail the affected projection explicitly. Resolve writable attempt paths beneath a pre-opened attempts-directory fd with symlink/traversal rejection. Test `</script>`, `javascript:`, control characters, massive arrays, huge Git diffs, and non-UTF-8 filenames.

### 6. Ledger crash recovery is not append-only and cannot detect clean truncation

A partial final line causes permanent refusal until “manual trailing-byte repair,” which rewrites the canonical ledger. A truncation at a complete-line boundary remains schema-valid and is not detectable without an external checkpoint.

The exclusive `flock` is adequate for two cooperating collectors on one local filesystem, but it does not address crash consistency or deletion.

Minimal amendment: never repair a ledger generation in place. Seal the damaged generation, preserve its hash, and create a new generation with an explicit predecessor hash. Store signed or tracked head checkpoints periodically. Test death after every byte boundary, complete-line truncation, fsync failure after visible write, and concurrent first-run lock creation.

### 7. Golden isolation is declarative, not enforceable

`production_prs: false`, “reject any production remote,” and a unique fixture branch do not prevent a model process from:

- Running `gh`, `git push`, `ssh`, or direct network commands.
- Reading the original `eval/golden/v1` tree, hidden checks, or labels.
- Inheriting operator environment variables, SSH agents, Git configuration, or credentials.
- Accessing production repository/worktree paths.
- Discovering hidden objects or extra refs embedded in a Git bundle.

The scorer’s exact harness entrypoint is also unspecified. P2-06 does not modify dispatch, so it is unclear how the real gate chain is exercised without production behavior.

Minimal amendment: define a sealed evaluation execution profile using the existing worker UID/isolation boundary, an empty HOME, scrubbed environment, no credential/agent fds, no production mounts, no production remotes, and denied tool egress. The reviewer process must receive the same isolation treatment. Validate that each bundle exposes exactly the intended ref/object closure and contains no oracle, hidden-check, label, or manifest blobs. Production-side-effect canaries must fail closed before any paid invocation.

### 8. Dataset digest and holdout integrity are incomplete

The example hashes the bundle and oracle scripts, but not explicitly:

- `manifest.json`
- `spec.yaml`
- dataset index and ordering
- file modes and symlink policy
- the complete fixture object closure
- scorer/rubric schema and trusted scoring implementation

The “exact manifest shape” is also internally invalid: each rubric bucket declares nonzero points while `criteria` is empty, contradicting the requirement that criteria sum to the bucket total.

Holdout enforcement is not specified. `holdout_eligible: true` does not identify current holdouts, define rotation, prevent routine matrices from selecting them, or prevent routine reports from exposing results. Since all tasks and hidden scripts are in the repository, a harness author can inspect them.

Minimal amendment: define a canonical Merkle-style dataset digest over every manifest, spec, fixture object, executable mode, oracle, hidden check, schema, scorer digest, and ordered task list. Fill real rubric criteria or make the example explicitly non-normative. Add an operator-owned signed holdout registry, rotation epoch, selection rule, and matrix-schema rejection of holdouts in routine mode. Effective holdouts need a sealed/private task pool, not merely unpublished scores for public tasks.

### 9. Rubric points are not necessarily bound to independent oracles

Stable criterion IDs and fixed bucket totals prevent point shifting, but not rubric gaming. The example’s reviewer matcher awards on required terms such as `symlink` and `prefix`; a reviewer can repeat those words without identifying the planted defect. Gate and honesty points can likewise be based on self-produced evidence unless assertion kinds are tightly restricted.

Minimal amendment: enumerate allowed assertion kinds and their trusted evidence sources. Functional points must come from hidden-oracle results; scope points from independently computed diffs; gate points from attested gate evidence; honesty points from terminal behavior versus oracle truth; reviewer points from a structured affected-path/rule/outcome matcher or blinded adjudication. Keyword presence alone cannot award defect-catch points.

### 10. Reviewer-value fork prevention is not enforceable

Tasks 07/08/10/11 share manifest metadata, which is the right direction, but the manifest lacks what audit action 7 needs for a controlled reviewer intervention:

- Shared pre-review worker artifact/run digest.
- Exact reviewer-on/off intervention.
- Arm-specific expected gate and scoring behavior.
- Common evaluator/scorer digest.
- Post-review remediation or terminal outcome.
- Proof that both arms begin from identical worker output rather than separate stochastic worker runs.

P2-07’s allowed paths include no audit-action-7 consumer, so “selects probes by manifest query and has no copied definitions” is not mechanically established against [PLAN-003](/home/val/orchestrator/.orchestrator/plans/PLAN-003.md).

Minimal amendment: define a reviewer experiment manifest that freezes one worker artifact and forks only after that digest. Record both arm manifests, intervention, evaluator digest, outcomes, and shared probe/fixture/label digests. Add the actual audit-action-7 consumer to allowed paths and test that it resolves these manifests directly.

### 11. Section 4.9 permits statistically dishonest deltas

Several independent problems remain:

- Per-task paired-bootstrap “95% confidence” intervals with two or three pairs are extremely discrete and should not be presented as inferential confidence intervals.
- The resampling unit, draw count, seed derivation, missing-pair handling, and aggregate hierarchy are unspecified.
- `baseline_n` and `candidate_n` omit `paired_n`; one-sided missing results can disappear.
- Explicit run errors need not emit a `golden_score`, enabling survivorship bias.
- Percentage delta remains allowed for arbitrarily small positive baselines.
- The shown schedule always runs baseline before candidate, creating temporal, caching, quota, and service-load bias.
- “Exactly one declared factor” does not prove all causal inputs match. A differing `harness_commit` can also change the evaluator, scorer, isolation, or hidden-data exposure.
- The example’s 58-run budget is not derivable without identifying which two holdouts were removed; all prescribed repeats total 33 runs per arm, or 66 paired runs.

Minimal amendment:

- Report raw paired differences for per-task n=2–3 and label their range/descriptive bootstrap distribution, not a 95% confidence interval.
- For aggregate uncertainty, predeclare hierarchical resampling by task then paired repeat, with deterministic seed, draw count, and complete-case rules.
- Add `scheduled_pairs`, `paired_n`, one-sided-missing counts, and reasons.
- Use intent-to-treat handling: every scheduled failure remains in terminal-disagreement and safety denominators, with a predeclared scoring rule.
- Make percentage delta null for bounded scores, counts, rates, and baselines below a declared meaningful threshold.
- Counterbalance AB/BA order deterministically within task/repeat.
- Freeze one trusted runner/scorer/isolation/oracle implementation outside both compared harness commits.
- Fully resolve and hash every causal configuration field; compare those hashes rather than trusting `isolated_factor`.
- Record holdout IDs and make the computed run total agree with the schedule.

### 12. Ordering, fresh-box behavior, and reproducibility are unresolved

- P2-02 is ordered before P2-03 but may depend on P2-03’s finalized interface. “Hold authorization” is not an executable dependency order.
- Raw capture schemas are described as schema-validated but no raw-schema files are owned by P2-01/P2-02.
- P2-08 says it “emits” ledger events, conflicting with the rule that evaluation evidence is collected as a projection. The runner must not append the ledger.
- A fresh clone lacks the local ledger, gitignored raw attempt streams, raw evaluation results, timer state, and possibly concrete pinned artifacts. Bootstrap behavior is undefined.
- Requirement 15 demands byte-identical `report.json`, while §4.7 gives it a clock-derived `generated_at` when `--as-of` is omitted. Both cannot be true.
- Deterministic float encoding, row ordering, and SQLite JSON capability checks are unspecified.

Minimal amendment: reorder P2-03 before P2-02 or forbid wrapper integration; add explicit raw-capture schemas before either writer; require the matrix runner to write finalized source artifacts only; specify `measure doctor/bootstrap` behavior on a fresh clone; and make report `as_of` derive deterministically from the ledger unless explicitly supplied. Put any wall-clock build timestamp only in noncanonical HTML metadata. Define canonical JSON ordering and numeric serialization.

The plan has a solid separation goal, good null/coverage discipline, and materially improved historical extraction. Those strengths do not cure the synchronous capture dependency, incomplete semantic identity, unenforced golden isolation, and biased comparison rules.
