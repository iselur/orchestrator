# PLAN-003 — Phase 1 verdict integrity and metrics semantics

## 1. Decision & non-goals

Phase 1 will establish a non-self-gating path before changing the existing gate. A new Dispatch 0 will install a minimal frozen bootstrap verifier outside candidate-controlled repository content. That verifier will carry a frozen required-test inventory, a frozen expected-assertion-ID inventory, and the minimal strict approval parser/schema required for its own approvals; run candidate code through the worker boundary; provision root-owned trusted snapshots under `/srv/codexwork/installed/<commit>`; and issue machine-checked activation manifests. Its root-owned `bootstrapctl` launcher/service will be the exclusive pre-Dispatch-4 authorization choke point: only that path can reach authoritative attempt-evidence roots or the PR/push credential. It will gate Dispatches 1–4 and validate Dispatch 4’s activation; Dispatch 4 may not activate by selecting and passing only its own new policy or tests.

After Dispatch 0:

- Dispatch 1 performs a high-assurance, behavior-preserving metrics extraction.
- Dispatch 2 performs the ordinary-risk metrics semantic correction.
- Dispatch 3 performs the high-assurance assertion migration.
- Dispatch 4 installs the repaired parent-owned test-attestation gate, isolated environment, source/target binding, tree revalidation, activation enforcement, and reviewer evidence.
- Dispatch 5 installs verdict schema v4, positional criterion binding, duplicate-key rejection, and structured blocking findings under the activated Dispatch 4 parent.

The installed parent dispatcher—not candidate content—will select required tests from the exact approved base and candidate Git trees. Each required test will have a parent-owned exact expected-assertion-ID set. Authorization requires an exact set match: every expected ID exactly once, with no missing, unknown, or duplicate assertion.

Trusted assertion events will be producer-authenticated operationally. The event-producing controller runs on the dispatcher/operator side, outside the `codex-worker` candidate UID and candidate process namespace. Candidate application processes receive neither an attestation nonce nor an assertion descriptor. Parent-owned command helpers scrub attestation and Python environment variables, close all non-whitelisted descriptors before every candidate `exec`, and use per-invocation nonces and separate trusted and supplemental pipes. The trusted controller alone converts observed command outcomes into assertion events and owns aggregation. If the UID/process separation cannot be established, the test phase aborts before candidate launch.

Required test content, runner, policy, helpers, fixtures, schemas, wrappers, and interpreter identities will come from a root-owned, read-only installed snapshot materialized from the exact parent commit. Candidate-modified tests remain supplemental. The candidate tree will be revalidated after the test phase, immediately before review, and immediately before PR creation.

Review verdict schema v4 will retain positional binding: one ordered entry for every acceptance criterion persisted in the attempt manifest before candidate work. Review JSON parsing will reject duplicate object keys and nonstandard constants. Structured scope, regression, and security blockers will mechanically prevent PASS.

Metrics will retain the extraction/semantics split. Extraction parity baselines will be captured by the operator before candidate work and bound by the approval artifact. Straight-through metrics will be anchored by terminal-spec time while consulting the spec’s complete attempt history.

Phase 2 implementation work requires all six Phase 1 dispatches—Dispatches 0–5—to be completed, merged where applicable, installed, activated, and validated. Dispatch 2 is necessary but not sufficient.

PLAN-003 embeds the minimal strict approval parser/schema needed by Dispatch 0 in the independently reviewed, operator-installed frozen artifact. The parser, schema, and their exact digests are approval-bound, and Dispatch 0 refuses any schema-digest mismatch. The separate approval-schema plan is explicitly removed from PLAN-003’s critical path; it may upgrade the full native system later, after the non-self-gated bootstrap path exists.

Non-goals:

- Do not create a complete second production gate in Dispatch 0. It is a deliberately minimal, frozen bootstrap verifier.
- Do not require a read-only bind mount of the candidate tree in revision 4. Tree revalidation is mandatory; a read-only bind is optional hardening.
- Do not duplicate the full sealing, terminal-manifest, and evidence-hash-chain design assigned to the separate `evidence-hashing` plan.
- Do not claim that escaping or delimiting reviewer log text neutralizes semantic prompt injection.
- Do not add a hash-locked wheelhouse or redesign dependency locking in this plan.
- Do not change the worker model, reasoning depth, priority service tier, reviewer seat, branch topology, or human merge authority.
- Do not permit candidate dispatcher, runner, policy, schema, helper, fixture, or test content to authorize itself.
- Do not treat aggregate exit zero, a positive assertion count, free-form `test_command` output, or reviewer prose as proof that required tests ran.
- Do not expose the operator’s home or `.venv`, install packages as `codex-worker`, enable candidate network access, or import Python modules from the candidate root into the trusted controller.
- Do not alter `.orchestrator/attempts/SPEC-015/1/*`.
- Do not retroactively authorize legacy verdicts or test logs.
- Do not infer an unmatched merge’s actor.
- Do not authorize an incompatible test change outside the three-state compatibility sequence or a separately approved external bootstrap.
- Do not allow CI’s portable suite to substitute for an on-box activation artifact.
- Do not allow direct pre-Dispatch-4 invocation of repository `scripts/dispatch` to create authoritative evidence, reach PR/push credentials, or select an active parent.
- Do not make the separate full-system approval-schema plan a prerequisite for Dispatch 0.

PLAN-003 must implement the B6 subset assigned here, use Dispatch 0’s embedded strict approval parser/schema for bootstrap authorization, and depend on the separate evidence-hashing plan for the remaining sealing work.

Acceptance criteria:

1. **R9.1:** Every new attestation aggregate, criteria vector, review request, review result, and activation-related attempt artifact is created once with `O_CREAT|O_EXCL` or an equivalent no-replace primitive.
2. **R9.2:** Final evidence paths and parent directories are dispatcher-owned and not writable by the worker or candidate.
3. **R9.3:** Worker output crosses bounded pipes; candidate processes receive no writable final evidence pathname.
4. **R9.4:** Existing destinations, symlinks, hardlink aliases, rename swaps, concurrent writers, and resume-after-terminal attempts cannot replace a finalized aggregate, log, or review result.
5. **R9.5:** A finalized attempt is never reopened for authorization writes. Resume requires a new attempt number or explicit non-authorizing display access.
6. **R9.6:** SPEC-015 bytes and hashes remain unchanged after the real replay.
7. **R9.7:** Full artifact sealing, directory `fsync` policy, terminal hash manifests, and comprehensive retained-evidence integrity remain assigned to the separate `evidence-hashing` plan.
8. **R9.8:** Dispatch 0’s embedded strict approval parser/schema validates exact plan revision, dispatch, attempt, candidate digest, approval-authorized genesis anchor, Dispatch 0/verifier/inventory/schema digests, parity-baseline digest, expected test/assertion inventory, and every changed test’s old/new blob identity. The separate approval-schema plan may upgrade the full native system later but is not a PLAN-003 prerequisite.
9. **R9.9:** Dispatch 0 measures its embedded parser and schema from the installed frozen artifact and refuses before evidence creation, candidate launch, activation, credential access, push, or PR creation if either measured digest differs from the digest bound by the approval.

### R10. Compatibility, dependency, and Phase 2 sequence

Test evolution and dependency upgrades must not rely on self-promotion.

Acceptance criteria:

1. **R10.1:** An ordinary compatible test implementation change passes both the installed parent test and the candidate supplemental test before later promotion.
2. **R10.2:** A behavior change that contradicts the old strict test uses the complete three-state sequence in §4.7; each stage is a separate normal high-assurance dispatch and activation.
3. **R10.3:** An end-to-end test exercises all three states and proves that no stage’s candidate test is authoritative during its own attempt.
4. **R10.4:** Phase 2 implementation dispatch refuses until Dispatch 0 is installed and Dispatches 1–5 are merged, installed, activation-manifested, selected through valid atomic head transitions, and validated.
5. **R10.5:** If a compatibility bridge is impossible, the normal pipeline fails closed pending separately scoped operator bootstrap authority; this plan grants no such authority.

## 4. Design / approach

### 4.1 Dispatch decomposition and dependency order

Dispatch 0’s independently reviewed frozen artifact includes the minimal strict approval parser/schema required to validate its own digest-bound approvals. The separate approval-schema plan is explicitly removed from this sequence’s critical path and may upgrade the full native system later.

| Order | Dispatch | Lane | Dependency | Purpose |
|---:|---|---|---|---|
| 0 | Bootstrap verifier, authorization choke point, trusted snapshots, and activation state machine | operator-installed high-assurance bootstrap | Independently authorized frozen artifact and approval-authorized genesis anchor | Install the frozen non-candidate verifier, embedded approval parser/schema, inventories, exclusive evidence/credential launcher, append-only activation ledger, and CAS head; provision `/srv/codexwork/installed/<commit>`; gate Dispatches 1–4; create machine-checked activation state. |
| 1 | Metrics extraction | high-assurance | Dispatch 0 installed; current parent selected as active head | Extract reporting while preserving approval-bound baseline behavior. |
| 2 | Metrics semantics | ordinary | Dispatch 1 merged, frozen-verified, installed, activated, and selected | Correct attribution and straight-through semantics without dispatcher write scope. |
| 3 | Assertion-event migration | high-assurance | Dispatch 2 merged, activated, and selected | Migrate tests to exact assertion IDs, exit-77 diagnostics, source/target separation, and per-assertion mutation proof. |
| 4 | Attestation, trusted execution, venv, activation enforcement, tree binding, and reviewer evidence | high-assurance | Dispatch 3 merged, frozen-verified, installed, activated, and selected | Install the real parent gate under the frozen verifier and make activation enforcement native. |
| 5 | Verdict v4 and blocking findings | high-assurance | Dispatch 4 activated by Dispatch 0 artifact, atomically selected as head, and accepted by native activation check | Add strict JSON parsing, launch-time criteria vector, positional v4 binding, and blocker evaluation. |

“Completed” means merged where repository content exists, exact installed snapshot provisioned, no intervening commit accepted, required verifier passed, immutable activation manifest and ledger transition accepted, and the exact commit selected by the singleton active head.

### 4.3 Dispatch 0 bootstrap verifier and exclusive authorization path

Dispatch 0 is a small frozen verifier, not a duplicate production dispatcher. Its inventory has four immutable profiles, one for each of Dispatches 1–4. Each profile specifies:

- exact allowed repository paths;
- exact bootstrap test IDs;
- exact expected assertion IDs per test;
- immutable test/helper content hashes;
- candidate command entry points;
- required worker-boundary properties;
- activation drills required after merge;
- expected predecessor active commit and head transition;
- required snapshot-closure components.

The frozen artifact also contains the minimal strict approval parser/schema needed by Dispatch 0. Independent authorization binds the exact parser and schema digests together with the verifier, launcher, inventory, plan revision, genesis anchor, and required dispatch-specific identities. `bootstrapctl` measures these files from the root-owned installed artifact and performs strict duplicate-key-rejecting validation before it requests any privileged operation. A parser/schema digest mismatch fails before evidence allocation or credential access. The separate approval-schema plan is neither invoked nor assumed for Dispatch 0.

The verifier runs from `/srv/codexwork/bootstrap`, never from the candidate checkout. It invokes candidate components as data through the `codex-worker` boundary. Its assertions are computed by the frozen controller from observed results; candidate JSON does not count.

For Dispatches 1–4, `bootstrapctl` is the exclusive operator-facing authorization path. A root-owned service behind it alone can allocate a directory in the authoritative attempt-evidence root, append activation records, compare-and-swap the active head, and use the PR/push credential. It supplies a single-use, attempt-bound launch manifest/capability only after verifying the approval, installed parent, selected head, inventory, candidate identity, and allowed dispatch. The repository dispatcher and candidate processes never receive the PR/push credential itself; a successful PR/push request is mediated by the service and bound to the already authorized commit and attempt.

Direct pre-Dispatch-4 invocation of repository `scripts/dispatch` has no valid service-issued launch manifest/capability, no writable authoritative evidence directory, and no credential-broker authorization. The mandatory authorization preflight therefore returns nonzero before worktree creation, candidate execution, reviewer invocation, push, or PR creation. Files created outside the authoritative evidence root are explicitly non-authorizing and cannot be imported or promoted into it. The bypass regression invokes the repository path directly under the ordinary operator and worker contexts and requires attempt-evidence, candidate-launch, reviewer, push, and PR spies to remain at zero.

After a candidate is merged, `bootstrapctl` verifies the exact integration commit, rejects intervening commits, provisions its installed snapshot, runs the frozen post-merge profile, creates the immutable activation manifest and durable ledger transition, and atomically compare-and-swaps the singleton head.

Dispatch 4 must pass both its candidate-phase frozen profile and its post-merge frozen activation profile. Its own new tests are supplemental to this activation decision.

### 4.4 Installed snapshot and race-safe closure

Snapshot provisioning materializes the exact Git commit into `/srv/codexwork/installed/<commit>` without using a mutable checkout as the final source. The provisioner:

1. Resolves the commit and tree objects.
2. Enumerates the full gate closure.
3. Copies each expected blob into a new root-owned staging directory using no-follow creation.
4. Records repository path, mode, object ID, content hash, size, and role.
5. Records verified identities for Bash, trusted Python, and any other interpreter.
6. Removes worker write access recursively.
7. Atomically installs the previously absent commit directory.
8. Verifies the installed closure against the manifest before activation.

The native runner pins a directory descriptor and resolves closure components beneath it using no-follow, beneath-only semantics. Native executable components use already-open descriptors where supported. Required shell tests are passed to the verified Bash interpreter through an already-open script descriptor; helpers and fixtures are pre-opened or copied from verified descriptors into a dispatcher-owned, non-worker-writable invocation bundle. No verified candidate-writable pathname is later reopened for execution.

Concurrent replacement tests target files and directory components, including helpers, fixtures, schemas, and interpreter shadows. Worker inability to write the root-owned snapshot is necessary but does not replace closure verification.

### 4.5 Canonical Git change model and required-test policy

Policy evaluation compares the complete path maps of `base_sha^{tree}` and `worker_commit^{tree}`. Each path state includes Git mode, object ID, and object type. Rename detection may be recorded for display, but requiredness is calculated from the underlying additions, deletions, and modifications, so rename thresholds cannot suppress coverage.

The policy must cover:

- regular-file content and mode changes;
- symlink targets and regular/symlink type changes;
- executable-bit changes;
- gitlinks/submodules and `.gitmodules`;
- additions and deletions;
- both sides of displayed renames;
- unknown paths at repository root;
- generated control-plane configuration;
- unrecognized paths below `scripts/`, `tests/`, `specs/`, `.github/workflows/`, and `.orchestrator/`.

Every changed path yields a nonempty required-test set or a policy error. Candidate policy changes never affect their own attempt.

Worktree state is a separate transition invariant. Untracked or dirty control-plane content fails even if it is absent from the committed tree.

### 4.6 Assertion-producer privilege boundary and attestation protocol

The event-producing controller runs on the dispatcher/operator side under a UID distinct from `codex-worker` and outside the candidate service/process namespace. Candidate programs and candidate-controlled test targets run as `codex-worker`. Before any candidate launch, the root-owned launcher verifies the controller UID, candidate UID, service/cgroup or namespace placement, dumpability and process-access policy, descriptor table, and absence of candidate authority over the controller. If the required separation or restriction cannot be established or verified, testing aborts fail-closed; “where supported” is not an authorization fallback.

The installed parent controller owns:

- the trusted and supplemental pipes;
- all per-invocation nonces;
- expected source class;
- exact expected assertion IDs;
- launch count;
- command outcome observation;
- aggregate construction;
- final evidence paths.

Trusted tests describe assertions and candidate commands through the parent-owned helper. For a candidate command, the controller/helper:

1. Validates the requested assertion ID against the active expected set.
2. Forks a candidate process through the worker boundary as `codex-worker`, outside the controller’s UID and process namespace.
3. Sets a fixed candidate-safe environment without nonce or attestation variables.
4. Scrubs `PYTHONPATH`, `PYTHONHOME`, `PYTHONSTARTUP`, and related variables.
5. Closes all descriptors except the explicit stdin/stdout/stderr and command-specific data descriptors.
6. Verifies that candidate-to-controller `ptrace`, `process_vm_readv`, `process_vm_writev`, and `pidfd_getfd` access is denied; inability to verify effective denial aborts before the candidate command.
7. Observes the command outcome.
8. Produces the assertion event itself.

The event schema is:

```json
{
  "version": 1,
  "run_nonce": "<controller nonce>",
  "source_class": "trusted",
  "test_id": "tests/example.sh",
  "assertion_id": "example.stable.condition",
  "sequence": 1,
  "event": "assertion",
  "ok": true
}
```

The exact assertion-ID set must equal the parent inventory set. Names or free-form text may be stored diagnostically but are not identities.

Trusted and supplemental invocations receive independent pipes and nonces. Supplemental results are stored only under `candidate_supplemental_results`; they can block on failure but can never satisfy `trusted_required_results`.

The aggregate records parent snapshot commit and manifest hash, policy hash, base and candidate tree IDs, expected and observed assertion-ID sets, invocation count, source class, controller and candidate UID/namespace identities, process-access-denial results, exit status, and overall status.

### 4.7 Candidate tree transitions and three-state compatibility

The candidate worktree is revalidated:

1. After all trusted, supplemental-test, and free-form candidate commands complete.
2. Immediately before reviewer invocation.
3. Immediately before PR creation.

Each check requires the expected `HEAD`, exact committed tree identity, clean index/tracked state, and no forbidden untracked content. Review and PR inputs use commit-object data.

A behavior change that contradicts the installed strict test uses three states:

1. **State A — bridge plus transitional test.** The old strict parent test remains authoritative. The candidate adds compatibility behavior and a transitional test that accepts the old and new behavior. Both old strict and supplemental transitional tests pass. After frozen/native verification, the transitional test becomes parent content.
2. **State B — strict-new test promotion.** With the transitional test authoritative and the bridge still present, a second dispatch adds the strict-new test. The supplemental strict-new test passes. After activation, the strict-new test becomes parent content.
3. **State C — bridge removal.** With the strict-new test authoritative, a third dispatch removes the compatibility bridge. The authoritative strict-new test must pass.

No candidate test is authoritative during its own attempt. If any state is impossible, the sequence stops and requires external bootstrap authority.

### 4.8 Activation manifest, ledger, and singleton head

The strict activation manifest contains:

```json
{
  "schema_version": 1,
  "commit": "<activated commit>",
  "previous_active_commit": null,
  "approval_authorized_anchor": "<genesis anchor commit>",
  "integration_tip": "<observed exact tip>",
  "approval_digest": "<approval artifact digest>",
  "dispatch0_sha256": "<exact frozen Dispatch 0 artifact digest>",
  "approval_schema_sha256": "<embedded schema digest>",
  "installed_snapshot": "/srv/codexwork/installed/<commit>",
  "snapshot_manifest_sha256": "<digest>",
  "verifier_sha256": "<bootstrap or native verifier digest>",
  "inventory_sha256": "<required test/assertion inventory digest>",
  "venv_id": "<ABI/platform-addressed identity>",
  "portable_validation": {"status": "PASS", "artifact_sha256": "<digest>"},
  "on_box_validation": {"status": "PASS", "artifact_sha256": "<digest>"},
  "transition_id": "<activation transition digest>",
  "created_at": "<timestamp>"
}
```

Genesis is the only form permitting `previous_active_commit: null`. In that form, `commit` must equal `approval_authorized_anchor`, the approval must authorize that exact anchor, and the manifest must bind the exact Dispatch 0 and embedded approval-schema digests. Non-genesis manifests require a commit-valued `previous_active_commit` and reject `approval_authorized_anchor`. The strict conditional schema rejects undeclared fields and duplicate keys.

Historical manifests are root-owned, non-worker-writable, and created only if absent. They do not alone make a commit active.

The root-owned activation state also contains:

- an append-only ledger of strict, no-replace transition records;
- a singleton `/srv/codexwork/activations/active-head.json`;
- a root-service transition lock used only to serialize compare-and-swap evaluation.

Each ledger record binds a unique transition ID, kind (`genesis`, `activate`, or `rollback`), expected prior head, target commit, target manifest digest, verifier and approval identities, integration-tip observation where applicable, and timestamp. The active-head record binds the selected commit, its manifest digest, and the ledger transition ID that selected it.

A transition proceeds as follows:

1. Acquire the root-service transition lock.
2. Strictly parse and verify the current head, its referenced ledger record, and its selected manifest, or prove that no head exists for genesis.
3. Compare the current commit and transition ID with the transition’s expected prior head.
4. Validate the target manifest, installed snapshot, verifier result, approval, integration relationship, and transition kind.
5. Create and durably record the immutable ledger entry using no-replace semantics.
6. Write and verify a new head record in a root-owned temporary file.
7. Atomically replace the singleton head while the checked prior head remains locked.
8. Durably record the containing directory before releasing the lock.

A stale compare fails without head replacement. Concurrent siblings therefore cannot both win. A crash before the durable ledger record leaves the prior head; a crash after ledger durability but before head replacement also leaves the prior head and an unselected ledger record; a crash after head replacement yields a head that references the already durable ledger record. Dispatch fails closed if the head, ledger, and manifest are inconsistent.

Every dispatcher validates its immutable historical manifest and proves that its installed commit is the singleton selected head before attempt creation. A stale parent refuses even though its historical manifest remains valid.

Rollback is a new approved `rollback` transition from the current selected head to a previously manifested, installed, closure-verified target. It appends a ledger record and compare-and-swaps the head; it never edits or deletes historical state.

### 4.9 Verdict v4 criterion normalization and binding

At attempt launch, the dispatcher persists the exact ordered criteria vector from the digest-approved spec. It records raw text, normalized text, position, and vector digest. Review authorization never rereads the spec path.

Normalization remains:

1. Unicode NFKC.
2. CRLF and CR to LF.
3. Strip leading and trailing Unicode whitespace.
4. Replace each remaining maximal Unicode-whitespace run with one ASCII space.

There is no case folding, punctuation removal, Markdown rendering, link resolution, fuzzy matching, or clause reordering.

Validation order is:

1. Size bound.
2. Strict JSON parse with duplicate-key and nonstandard-constant rejection.
3. Schema v4.
4. Artifact and activation-parent bindings.
5. Criteria-vector digest.
6. Exact count.
7. Sequential indices.
8. Positional normalized text equality.
9. Binary review evaluation.

### 4.10 Structured findings and blocker predicate

The retained blocker predicate is:

```text
blocking =
    findings is not an array
    OR any entry is not an object
    OR any required field is absent or invalid
    OR any undeclared field is present
    OR severity is not exactly "advisory" or "blocker"
    OR any severity is exactly "blocker"
```

An empty array is the only “no findings” representation. `evaluate_binary_review()` rechecks the predicate even after schema validation.

### 4.11 Gate venv and staged dependency upgrade

The venv identity is a digest over a canonical record containing:

- hash of the trusted parent `scripts/requirements.txt` bytes;
- Python implementation and ABI/cache tag or SOABI;
- normalized OS/platform/machine tuple;
- provisioning-format version.

Trusted Python is invoked with `-I`, from a fixed safe cwd, with Python variables scrubbed. Direct dependency imports and their resolved origins are verified beneath the selected venv.

A candidate requirements change is staged:

1. The candidate is tested using the active parent’s dependency environment; candidate requirements are not installed.
2. After merge, the new commit is materialized as an inactive installed snapshot.
3. The previous frozen/native verifier provisions a new ABI/platform-addressed venv from that merged snapshot.
4. The verifier runs origin, isolation, and on-box tests.
5. Only then may an activation manifest and atomic head transition name the new venv and commit.

### 4.12 Reviewer evidence

The controller enforces the raw-log limit while reading process output. It never permits an unbounded file and hashes only the bounded captured artifact.

The reviewer receives:

1. Fixed instructions.
2. Machine-generated activation, tree, required-test, expected/observed assertion, and binding summaries.
3. A secondary JSON-escaped log tail labelled untrusted.
4. Fixed instructions after the evidence block.

Escaping protects syntax only. Mechanically decidable properties are resolved before reviewer invocation and are never delegated to model judgment.

### 4.13 Metrics extraction and definitions

Dispatch 1 parity compares against operator-captured, approval-bound baselines created before the candidate branch. The candidate may add regression tests but cannot redefine the baseline.

Dispatch 2 groups complete attempt history by spec. A spec’s canonical final terminal result determines its terminal timestamp. Window eligibility is based on that timestamp in `[start, end)`, while classification uses all retained attempts regardless of their timestamps.

- **Straight-through:** attempt 1 is the only attempt, reached normal `merged`, and retained evidence shows all normal gates passed.
- **Completed but not straight-through:** final terminal completion exists, but there was a retry, replacement, override, first-attempt failure, or later attempt.
- **In progress:** no canonical final terminal completion.
- **Malformed/incomplete:** safe classification is impossible.

A retry before the selected window still prevents straight-through status for an in-window completion. A later attempt after the window means the spec’s final terminal anchor is later and the earlier result does not independently enter the earlier denominator.

### 4.14 Scoped evidence handling and authorization authority

The dispatcher creates new authorization artifacts only at dispatcher-owned paths with no-replace semantics. Candidate output is consumed through pipes. Finalized attempts refuse new authorization writes.

Before Dispatch 4, only the root-owned `bootstrapctl` service can allocate or write the authoritative attempt-evidence root. A direct repository dispatcher has no authority to create or promote evidence there. The service retains the PR/push credential and mediates its use only for the exact commit and attempt authorized by the launch manifest after all deterministic gates and head-selection checks pass.

This plan intentionally stops short of defining the full evidence seal and hash-chain lifecycle. That work remains in the separate `evidence-hashing` plan.

### 4.15 Alternatives considered

1. **Continue with the existing gate until Dispatch 4.** Rejected because Dispatches 1–3 and Dispatch 4 activation would remain self-gated.
2. **Build a second complete production gate.** Rejected as unnecessary scope; Dispatch 0 is a frozen minimal verifier.
3. **Require merely one assertion.** Rejected because partial execution remains vacuous.
4. **Allow candidate assertion JSON with a nonce.** Rejected because inherited secrets and descriptors do not authenticate the producer.
5. **Make free-form `test_command` attestation evidence.** Rejected unless it is converted into a parent-owned ID/assertion contract; revision 4 treats it as non-evidentiary supplemental output.
6. **Require a candidate read-only bind mount.** Deferred as optional hardening; mandatory transition-time tree revalidation provides the scoped revision 4 invariant.
7. **Execute a verified shell script by reopening its pathname.** Rejected because it reintroduces a TOCTOU.
8. **Run trusted content from the operator’s home checkout.** Rejected because D5 intentionally makes the operator home unreachable to workers.
9. **Treat atomic rename as evidence immutability.** Rejected; new authorization artifacts use no-replace creation.
10. **Duplicate full evidence sealing here.** Rejected because it belongs to the separate evidence-hashing plan.
11. **Claim reviewer log escaping prevents prompt injection.** Rejected; it provides syntactic containment only.
12. **Address venvs only by requirements hash.** Rejected because Python ABI/platform changes can collide.
13. **Add full wheelhouse hash locking now.** Deferred to dependency/supply-chain work.
14. **Rely on rename detection for policy coverage.** Rejected; policy compares complete tree path maps.
15. **Make the separate approval-schema plan a prerequisite.** Rejected because validating and activating that repository change through the current failure-prone gate would leave a self-gated or circular trust dependency. Dispatch 0 instead embeds its independently reviewed minimal strict parser/schema; the full-system plan may land later.
16. **Reparse the spec at review time.** Rejected; the exact criteria vector is persisted at launch.
17. **Allow JSON duplicate keys and trust the last value.** Rejected as ambiguous authorization input.
18. **Unblock Phase 2 after metrics alone.** Rejected; Phase 2 requires all six Phase 1 dispatches.
19. **Use the revision 2 two-stage compatibility bridge.** Rejected as incomplete; revision 4 specifies the full three-state sequence.
20. **Declare `bootstrapctl` preferred while retaining parallel evidence and credential access.** Rejected because a convention does not remove the falsely-green dispatcher’s authorization path.
21. **Treat immutable per-commit activation manifests as the current-state selector.** Rejected because historical validity does not prevent stale parents or sibling activations; the append-only ledger and CAS singleton head provide selection.
22. **Run the event controller as `codex-worker` and rely on best-effort non-dumpability.** Rejected because same-UID process-access mechanisms remain; the controller/candidate UID boundary is mandatory and fail-closed.

=== ROUND-3 CRITIQUE (BLOCK, 3 findings) ===
Revision 4 remains blocked by three material defects.

1. The direct-dispatch preflight has no executable enforcement mechanism.

   Sections R1.3–R1.4 and §4.3 require direct invocation of the existing repository dispatcher to stop before worktree creation, candidate launch, or reviewer invocation. But observed fact 13 says that dispatcher is currently directly callable; Dispatch 0 does not modify repository content; and Dispatch 1 may modify `scripts/dispatch.py` only for metrics extraction. Protecting authoritative evidence and PR credentials prevents authorization, but it does not make the existing dispatcher perform the claimed early preflight or prevent it from launching workers and reviewers using the operator’s existing execution and Codex access. Section 4.3 even acknowledges that direct execution may create local files.

   Minimal amendment: define a concrete Dispatch 0 enforcement step that exists before genesis—for example, an independently authorized bootstrap patch adding the mandatory preflight to every repository entry point, combined with broker-only worker/reviewer execution authority and removal of ambient credentials. Specify which operation is guaranteed to occur before worktree creation and test old, copied, Python-direct, wrapper, operator, and worker entry points. Alternatively, weaken R1.4 to prohibit only authoritative effects, but that would abandon the explicitly required pre-candidate fail-closed invariant.

2. The mutable singleton head contradicts the stated preservation of prior head records.

   Section 4.8 atomically replaces `/srv/codexwork/activations/active-head.json` on every transition. Section 9.4 simultaneously says a prior head record is never edited or deleted and that originals remain evidence. No append-only head-generation artifact or archive operation is defined. The ledger contains similar state, but the document does not declare the replaced singleton to be a non-evidentiary derived projection whose historical bytes need not survive.

   Minimal amendment: either create an immutable, no-replace head-generation record for every transition and make the singleton a replaceable pointer to it, or explicitly state that only manifests and ledger records are historical evidence and remove the requirement to preserve prior singleton-head files. Add a test proving the chosen historical representation survives activation and rollback.

3. The crash test can pass while R1.18 is unmet.

   R1.18 requires exact phase-specific outcomes: before ledger durability the old head remains selected; after ledger durability but before replacement the old head remains selected; and after durable replacement the head references the already durable transition. Section 8.2 instead accepts “one consistent selected head or fail-closed recovery.” A missing or unusable head after either pre-replacement crash could therefore make the test green merely because dispatch refuses, even though R1.18 requires continued selection of the old head. Likewise, post-rename loss may be accepted as generic fail-closed recovery.

   Minimal amendment: make each crash injection assert the precise R1.18 state after an actual service restart or filesystem remount, including exact head bytes, referenced ledger record, manifest digest, and dispatchability of the required old or new selected parent. Reserve generic fail-closed recovery only for explicitly enumerated states outside those promised crash points.

The distinct-UID/process-namespace controller boundary and fail-closed syscall-denial checks are internally sufficient for the stated producer-authentication requirement. The remaining activation ledger/CAS sequencing is broadly coherent once the head-history contradiction and crash oracle are corrected.
