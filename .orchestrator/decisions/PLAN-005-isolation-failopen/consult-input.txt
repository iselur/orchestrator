# Close the isolation fail-open

## Decision

Treat execution isolation as a preflight policy decision, not a best-effort launch optimization.

`dispatch` must select one immutable execution mode before creating an attempt, branch, worktree, or Codex process:

| `isolation_available()` | Operator credentials | `ORCH_ALLOW_UNISOLATED` | Result |
|---|---:|---:|---|
| true | any | any | Use the existing `codex-worker`/systemd isolation path. |
| false | present or indeterminate | exactly `1` | Permit the same-user worker with a prominent warning; always run its TEST phase with a scrubbed environment and private network namespace. |
| false | present or indeterminate | unset or any value other than `1` | Refuse before launch with a nonzero exit. |
| false | absent | any | Permit the same-user worker with a warning; always sandbox its TEST phase as above. |

Additional decisions:

- `ORCH_ALLOW_UNISOLATED=1` overrides only the credential-based worker refusal. It must not disable TEST environment scrubbing or network isolation.
- A failure after selecting the isolated path must fail the attempt; it must never trigger a runtime downgrade to the same-user path.
- Any same-user path must first prove that its TEST network namespace can be created. If that proof fails, refuse before launching the worker.
- Only the literal string `1` activates the override. Values such as `true`, `yes`, or `01` do not.
- Credential-probe errors are indeterminate and therefore treated as credentials present.
- No secret values may appear in diagnostics or evidence.

## Goals

- Prevent a missing, broken, or sudo-denied D5 setup from silently exposing operator credentials to worker-authored code.
- Preserve the explicitly approved same-user escape hatch.
- Ensure every same-user TEST process receives a minimal, deterministic environment and cannot directly access the host IP network.
- Make downgrade, override, credential-detection, and sandbox failures observable and testable.
- Preserve the existing isolated worker and isolated TEST behavior.

## Non-goals

- Replacing or redesigning the `codex-worker` UID, systemd services, ACLs, worktree location, or `scripts/setup-worker-user.sh`.
- Restricting the worker’s network access; Codex still requires network access.
- Providing complete filesystem isolation for the same-user TEST fallback. The test can still address operator-readable paths by absolute pathname.
- Eliminating all local IPC routes such as pathname Unix sockets. The new boundary guarantees an isolated IP network namespace and closed inherited descriptors, not full host containment.
- Solving the separate CI-skip, hard-coded operator-home, sudoers, resource-limit, approval, evidence-integrity, or merge-authority findings.
- Changing spec schema, branch policy, review policy, test commands, or gate ordering.
- Making `ORCH_ALLOW_UNISOLATED=1` a persistent configuration or normal operating mode.
- Adding a Python package dependency; the fallback may depend on the host’s trusted `util-linux` `unshare` executable.

## Affected trust boundaries and interfaces

### Operator-to-worker boundary

`scripts/dispatch.py` becomes responsible for proving one of:

- D5 isolation is available; or
- same-user execution is allowed because no configured credential indicator was found; or
- the operator deliberately supplied the exact break-glass override.

The decision must occur before untrusted worker code can execute.

### Worker-to-TEST boundary

Worker-authored repository code is untrusted during TEST. On the same-user path, TEST must execute through one dedicated launcher that:

- starts from an empty environment;
- uses a temporary HOME and XDG directories;
- enters a new user and network namespace;
- closes all unrelated inherited file descriptors;
- invokes a non-interactive shell without profile files;
- never retries outside the namespace.

### Public operator interface

`ORCH_ALLOW_UNISOLATED=1` is the sole new/confirmed break-glass interface. Document it in `CLAUDE.md` as invocation-scoped, security-sensitive, and ineffective at bypassing TEST hardening.

Refusal diagnostics must state:

- isolation is unavailable;
- credential indicators were found or could not be safely assessed;
- no worker was launched;
- D5 setup should be restored with `scripts/setup-worker-user.sh`;
- `ORCH_ALLOW_UNISOLATED=1` is the explicit emergency override.

### Internal interfaces in `scripts/dispatch.py`

Keep the implementation centralized in `scripts/dispatch.py` rather than adding a second launcher module. Introduce testable top-level units equivalent to:

- `assess_operator_credentials() -> CredentialAssessment`

- `select_execution_mode() -> ExecutionDecision`
- `run_same_user_test(...)`

`ExecutionDecision` must distinguish `ISOLATED` from `SAME_USER` and retain whether an override was activated and the non-secret credential indicator categories used in the decision.

## Detailed design

### Credential assessment

Resolve the operator’s authoritative home from `pwd.getpwuid(os.geteuid()).pw_dir`; do not trust `$HOME` alone. Also inspect `$HOME` when it resolves to a different location and any explicitly configured credential roots such as `CODEX_HOME` or `GH_CONFIG_DIR`.

Classify credentials as present when any of the following exists:

- A nonempty environment variable whose name contains a credential term such as `TOKEN`, `SECRET`, `PASSWORD`, `PASSWD`, `API_KEY`, `PRIVATE_KEY`, or `CREDENTIAL`.
- A nonempty credential-capable pointer or agent variable, including `SSH_AUTH_SOCK`, `GPG_AGENT_INFO`, `DBUS_SESSION_BUS_ADDRESS`, `AWS_SHARED_CREDENTIALS_FILE`, `AWS_WEB_IDENTITY_TOKEN_FILE`, `GOOGLE_APPLICATION_CREDENTIALS`, `KUBECONFIG`, `DOCKER_CONFIG`, `CODEX_HOME`, or `GH_CONFIG_DIR`, when applicable.
- A known credential artifact, including:
  - `.codex/auth.json`
  - `.config/gh/hosts.yml`
  - `.git-credentials`
  - `.netrc`
  - `.aws/credentials`
  - `.config/gcloud/application_default_credentials.json`
  - `.azure/accessTokens.json`
  - `.docker/config.json`
  - `.kube/config`
  - `.npmrc`
  - `.pypirc`
  - non-public-key material under `.ssh`
  - entries under `.gnupg/private-keys-v1.d`, `.local/share/keyrings`, or `.password-store`
- A known credential path that is a symlink, even if broken.
- A stat, enumeration, ownership, or permission error that prevents a conclusive result.

Centralize these probes in one constant registry. Tests must cover each category. Probe metadata only; never open or log credential contents. Diagnostics may report safe indicators such as `env:OPENAI_API_KEY` or `file:.codex/auth.json`.

This predicate is intentionally conservative. A false positive requires D5 or the explicit override; a false negative may expose unknown credential forms and is therefore the more serious error.

### Execution-mode selection

Perform preflight after parsing and validating the requested spec but before mutable attempt setup.

1. Call `isolation_available()` once for mode selection.
2. If true, return `ISOLATED`; ignore the override because it was not needed.
3. If false, assess credentials.
4. If credentials are present or indeterminate and the override is not exactly `1`, raise a dedicated preflight error.
5. Prove that the same-user TEST namespace can be established.
6. If that proof fails, raise a preflight error regardless of override.
7. Return `SAME_USER`, retaining whether the override was used.
8. Recheck the credential assessment immediately before a same-user worker subprocess is created. Abort if the result became unsafe and the snapshotted override was not active.
9. Once `ISOLATED` was selected, any systemd launch error is terminal for that attempt. Remove every catch block or fallback branch that could downgrade it.

All launch paths, including retry or remediation paths, must consume this decision rather than independently calling `isolation_available()`.

### Same-user TEST environment

Build the TEST environment from an empty mapping. Do not copy `os.environ` and delete known secrets.

Set only:

- A fixed system `PATH`, such as `/usr/local/sbin:/usr/local/bin:/usr/sbin:/usr/bin:/sbin:/bin`.
- `HOME` pointing to a new mode-0700 temporary directory.
- `TMPDIR`, `XDG_CONFIG_HOME`, `XDG_CACHE_HOME`, `XDG_DATA_HOME`, and `GNUPGHOME` pointing to private empty subdirectories.
- `LANG=C.UTF-8`, `LC_ALL=C.UTF-8`, `TZ=UTC`, and `CI=1`.
- `GIT_CONFIG_GLOBAL=/dev/null` and `GIT_CONFIG_NOSYSTEM=1`.

Do not propagate `ORCH_ALLOW_UNISOLATED`, cloud variables, proxy variables, shell startup variables, language startup hooks, dynamic-loader variables, agent sockets, GitHub variables, or any arbitrary spec-provided environment.

Invoke the test through absolute trusted executables and `/bin/bash --noprofile --norc -c <test_command>`. Use the worktree as `cwd`, stdin from `/dev/null`, captured stdout/stderr, `close_fds=True`, and `umask 077`. Remove temporary directories in `finally`, including interrupt paths.

### Same-user TEST network isolation

Use the trusted system `unshare` executable from `/usr/bin/unshare` or `/bin/unshare`, rejecting a candidate that is not root-owned or is group/world-writable.

The effective command shape is:

```text
env -i <allowlisted assignments>
unshare --user --map-current-user --net --fork
/bin/bash --noprofile --norc -c <test_command>
```

The new network namespace contains no host interfaces or routes. Loopback may remain down; tests must not depend on it.

Before launching the worker in same-user mode, run a short, scrubbed `unshare ... /bin/true` capability probe. The real TEST invocation must still use `unshare`; the probe is not a substitute. If the real namespace setup fails, record a TEST infrastructure failure and do not retry the command directly.

Do not implement network denial with proxy-variable removal, DNS changes, firewall mutations, or an internet reachability probe. Those approaches are either bypassable, host-global, privilege-dependent, or nondeterministic.

### Diagnostics and evidence

For a refusal before attempt creation, emit a concise stderr diagnostic and return a distinct nonzero preflight status. Do not create a branch, worktree, attempt directory, or raw worker log merely to record the refusal.

For launched attempts, include non-secret execution metadata in the existing launch/attempt evidence:

- `execution_mode`: `isolated` or `same_user`
- `isolation_available`: boolean
- `unisolated_override_used`: boolean
- `credential_assessment`: `absent`, `present`, or `indeterminate`
- credential indicator names/categories, never values
- `fallback_test_network_isolation`: `not_applicable` or `unshare-netns`

If adding these fields would violate a currently validated evidence schema, update that schema in the same spec rather than silently omitting or overloading an existing field.

## Alternatives considered

- **Always refuse unisolated execution:** Stronger and simpler, but rejects the task’s deliberate allowance for credential-free environments and the explicit operator override.
- **Retain automatic fallback with a warning:** Rejected because warnings do not protect credentials and preserve the audited fail-open.
- **Make the override accept general truthy values:** Rejected because ambiguous parsing makes accidental activation more likely.
- **Scrub only known secret variables:** Rejected because new credentials and language/runtime injection variables would remain inherited. An empty environment plus allowlist is auditable.
- **Remove proxy variables without a network namespace:** Rejected because code can open sockets directly.
- **Use host firewall or iptables changes:** Rejected because they require elevated, host-global state and introduce cleanup and concurrency hazards.
- **Use an external internet endpoint as the network test:** Rejected because an offline endpoint produces a vacuous pass.
- **Fall back again when `unshare` is unavailable:** Rejected because that recreates the same fail-open in the TEST phase.
- **Downgrade after a systemd launch race or failure:** Rejected because preflight success does not authorize a later, weaker execution mode.
- **Move all launch logic into a new module:** Rejected for this bounded fix; central helpers in `scripts/dispatch.py` minimize scope while remaining testable.

## Failure modes and required behavior

| Failure mode | Required behavior |
|---|---|
| `sudo -n` denied, systemd missing, worker user absent, or another D5 prerequisite fails | `isolation_available()` is false; apply the decision table. |
| Credential probe cannot stat a path | Treat as indeterminate/present and refuse unless override is exactly `1`. |
| Override is misspelled or set to a truthy-looking value | Refuse; print the exact accepted form. |
| `unshare` is absent, untrusted, or denied by kernel policy | Refuse before worker launch, even with the override. |
| Isolated systemd launch fails after successful preflight | Fail the attempt; never execute as the operator. |
| Namespace creation fails during TEST | Fail TEST as infrastructure failure; never retry unsandboxed. |
| Test command exits nonzero inside the namespace | Preserve existing deterministic TEST-gate failure semantics. |
| Test attempts to read inherited token/proxy/agent variables | Variables are absent. |
| Test attempts direct IPv4/IPv6 connection or DNS resolution | Connection cannot reach the host or external network namespace. |
| Test relies on shell profiles or operator Git configuration | Profiles are disabled and global Git configuration is redirected. |
| Temporary sandbox cleanup fails | Report cleanup failure without changing the test result to success; leave no secret data because only allowlisted values entered the directory. |
| Credential detector produces a false positive | Operator restores D5 or deliberately uses the one-shot override. |
| Unknown credential form produces a false negative | Treat as residual risk; extend the centralized probe registry when identified. Do not weaken TEST hardening. |

## O4 — minor: preflight must precede the FIRST durable attempt artifact, named precisely

The draft says refusal must not create "a branch, worktree, attempt directory". At HEAD the attempt
directory + pinned verdict schema are written (~dispatch.py:701) and `claim_slot` records a durable
`launching` state row (dispatch.py:707) BEFORE the worktree exists. Step 4's "before mutable attempt
setup" must explicitly name: before att_dir creation AND before `claim_slot` — else an implementer
could place preflight after the slot claim and leak refusals into state as phantom launching rows.

## O5 — minor: name the second `isolation_available()` call site

`worktree_root()` (dispatch.py:540) calls `isolation_available()` independently of the launch-time call
(dispatch.py:717). Two calls can disagree (sudo timestamp expiry between them): launch.json records
`isolation:true` while the worktree lands under the same-user root, or vice versa. The draft's general
rule ("all launch paths consume this decision") covers it; make it explicit by listing dispatch.py:540
and :717 as the call sites to collapse into the single `ExecutionDecision`.

## O6 — minor/question: ambient-credential interaction with the new regression in CI

On any real host and in GitHub Actions the dispatcher's own environment contains credential indicators
(GITHUB_TOKEN, gh hosts.yml, .ssh, .codex/auth.json) — so the "credentials absent → same-user allowed"
cases of the new regression are reachable only under a constructed `env -i` fixture with fake HOME. The
draft implies this (temporary homes, fake runners) but should state it: every credentials-absent test
case must scrub the ambient environment AND point the passwd-home probe at a fixture home, else the
test is red on any machine that has real credentials — i.e. everywhere it matters.
